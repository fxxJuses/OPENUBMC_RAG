## 4. Code Review Summary

### H2 Changes (3 features in db93303)

1. **Structural query expansion**: Detects known component names and appends `service.json`/`model.json` keywords when structural triggers are present. Gated to semantic_match and mixed types only.
   - Implementation: Correct, well-gated with trigger conditions.

2. **Query type routing**: Adjusts BM25/Dense RRF weights by +/-0.1 based on query_type.
   - Implementation: Wired through `HybridSearchEngine.search()` and `RetrievalEvaluator`.
   - Problem: Weight shifts (+/-0.1) are too aggressive for the current weight balance (0.5/0.5). Moving to 0.4/0.6 or 0.6/0.4 causes borderline queries to fall below the file-matching threshold.

3. **Simple spelling correction**: Levenshtein distance <=1 for common openUBMC terms.
   - Implementation: Clean, no external deps. But the test dataset only has 1 case with a typo (TC-045 "sensr threshold config"), so this feature's impact is negligible in eval.

### Key Observations

- **The revert was correct.** File@5 dropped 8pp (0.58 -> 0.50), Recall@5 dropped 5pp (0.42 -> 0.37) — both exceed the 2pp regression threshold defined in the optimization plan.
- **single_component is the worst-hit category**: File@5 dropped 14.3pp (0.64 -> 0.50), Recall@5 dropped 10.7pp. The structural expansion adding "service.json"/"model.json" to component queries is likely hurting precision by flooding BM25 with generic terms that match too many files.
- **NDCG@5 improved significantly** (+7.5pp overall, +18.5pp for single_function). This means the H2 changes improve *ranking quality when hits occur*, but the weight routing pushes some correct files out of the top-5 entirely.
- **CategoryHit@5 dropped 10pp** (0.88 -> 0.78), meaning the correct repo/component is no longer found in 5 fewer cases — the query routing is misrouting some queries.
- **Easy queries unchanged**: All 0.58 baseline easy queries remain at the same values, suggesting spelling correction had no measurable effect (likely because easy queries already match correctly).

### Root Cause Analysis

The regression is primarily caused by **query type routing's weight adjustments**:

1. The 0.1 weight shift is too large. With base weights at 0.5/0.5, shifting to 0.4/0.6 means the disadvantaged retriever loses 20% of its influence.
2. The evaluator passes `query_type` from dataset labels to the search engine, but in production there is no such label — the auto-detect `_detect_query_type()` may classify differently.
3. The structural expansion adds "service.json"/"model.json" broadly when triggers are present, but these generic terms match across many components, diluting BM25 precision.

### Recommendations for next attempt

1. **Reduce weight adjustment**: Try +/-0.03 instead of +/-0.1. The dev-backend already tested +/-0.05 and +/-0.03 with "similar regression pattern" — this suggests the routing approach itself may need a different strategy (e.g., adjusting RRF k instead of weights).
2. **Gate structural expansion more aggressively**: Only expand when the query explicitly mentions a specific component + explicitly mentions "MDS" or "model definition" — not just generic triggers like "service" or "config".
3. **Consider routing at the Reranker stage instead of RRF fusion**: Adjust reranker bonuses by query_type rather than changing RRF weights, since the reranker operates on already-retrieved candidates and can't push files out of the candidate pool.
4. **Spelling correction is safe but low-impact**: Keep it in future iterations as a standalone, non-breaking improvement.
