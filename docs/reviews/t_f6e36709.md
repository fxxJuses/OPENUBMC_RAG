## 4. 根因分析

### 4.1 P3 分词修复回归 (-4pp)

虽然移除了复合 token（解决上一轮 -10pp 的主因），但**正则表达式的同步修改**引入了新的行为差异：

- `\b`（word boundary）在字母与 `_` 之间不匹配（因为 `_` 是 `\w` 字符）
- `[^A-Za-z]|$` 在字母与 `_` 之间匹配，导致 snake_case 标识符被完全拆分

这意味着 `IPMI_SEL_EVENT` 从一个整体 token 变成 3 个独立 token，改变了 BM25 的匹配行为。原基线索引使用 `\b` 正则，snake_case 标识符保持完整，对精确匹配更友好。

### 4.2 DashScope reranker 回归 (-7~-10pp)

DashScope qwen3-rerank 模型在此代码检索任务上表现不佳的可能原因：

1. **领域不匹配**：qwen3-rerank 是通用文本重排序模型，对嵌入式固件代码（OpenBMC/IPMI/Lua）的语义理解可能不足
2. **文档内容特性**：代码 chunk 包含大量标识符、API 调用、配置参数，与通用文本差异大
3. **score 替换**：DashScope reranker 直接用 API 返回的 `relevance_score` 替换原有分数，完全丢弃了 RRF+boosting 的精细排序信号
4. **top_n=20 过大**：对所有 50 个查询候选都进行 API 重排序，可能在低置信度区域引入噪音

### 4.3 两者叠加 (-14pp)

P3 正则修改改变了 BM25 索引的 token 结构，加上 DashScope 的非领域适配重排序，两者叠加产生最差结果。

## 6. 判定

**❌ REGRESSION — 两项改动均不通过**

### P3 分词修复
- File@5 = 54%（基线 58%，**Delta = -4pp**）
- 超出 ±2pp 容忍范围
- 根因：正则表达式 `\b` → `[^A-Za-z]|$` 的变更引入新的分词行为差异
- 建议：如需修复 `\b` 边界问题，应同时**重建基线索引**以保持索引-查询一致性，并重新确立基线

### DashScope qwen3-rerank
- File@5 = 44%（P3修复 + DashScope，**Delta = -14pp**）
- File@5 = 50%（原始基线 + DashScope，**Delta = -7pp**）
- 在任何索引条件下都造成退化
- 建议：
  1. DashScope reranker 默认禁用保持不变（`dashscope_reranker_enabled: false`）✅
  2. 考虑仅在 top_k 更小的窗口内应用（如仅对 top-5 重排序而非全部）
  3. 考虑将 DashScope 分数与原有分数加权融合，而非完全替换
  4. 考虑领域微调或选择代码专用 rerank 模型

## 7. 建议

### 短期（修复回归）
1. **恢复原始正则**：将 `_TOKENIZE_RE` 中的 `[^A-Za-z]|$` 改回 `\b`，消除 snake_case 拆分差异
2. **确认基线恢复**：用原始正则 + 重建索引验证 File@5 回到 58%±2pp
3. **DashScope 保持禁用**：代码质量合格但模型效果为负，不建议启用

### 中期（改进方向）
1. **P3 分词策略**：如需修复 `\b` 边界，应重建索引并重新跑基线，确保对比公平
2. **DashScope rerank 优化**：
   - 缩小重排序窗口（仅 top-10 候选）
   - 分数融合而非替换
   - 评估代码专用 reranker 模型
3. **清理死代码**：`_DOMAIN_DICTIONARY` 在简化后未被使用，应移除或集成到分词逻辑中
