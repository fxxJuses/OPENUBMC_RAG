### 注意：索引-查询分词不对称

当前的 BM25 索引在 P3 改动之前构建，索引端的 token 与查询端的新分词器不完全对称：
- 索引侧：旧分词器只有子 token（如 `get`, `sensor`, `data`）
- 查询侧：新分词器同时有复合 token + 子 token（如 `getsensordata`, `get`, `sensor`, `data`）

复合 token 在索引中无匹配，等同于噪音但不影响原有子 token 的匹配。这意味着评估可能略受影响，但方向偏保守（不会虚高）。

## 6. 代码审查要点

### P3 BM25 分词增强

**正面：**
- `code_tokenize()` 设计合理：保留复合 token 同时添加子 token，不丢失原有匹配能力
- 领域词典 `_DOMAIN_DICTIONARY` 和 `_DOMAIN_MULTI_WORD` 实用性强
- `preserve_composites` 参数保证向后兼容
- regex 修复正确：`\b` 确实无法匹配字母→下划线边界

**关注点（非阻塞）：**
- `_DOMAIN_MULTI_WORD` 的 greedy matching（从最长到最短）正确，但性能在极大文档上可能有问题（对每个 token 位置最多尝试 5 种长度）
- `char_to_token` 映射使用 `text.index(t, idx)` 可能在重复 token 时定位不准，但对当前场景足够

### P0 交叉编码器

**正面：**
- 延迟初始化设计好：不使用时不加载模型，零开销
- 启发式 fallback 完整：`sentence-transformers` 不可用时自动降级，不会 crash
- 异常处理完善：模型加载失败、推理失败都有 fallback
- `skip_cross_encoder` 参数提供了细粒度控制

**关注点（非阻塞）：**
- 启发式 fallback 的 70/30 权重（`candidate.score * 0.7 + overlap_score * 0.3`）是硬编码的，后续可考虑可配置化
- `_model_rerank` 中 `source="cross_encoder"` 而 fallback 中 `source="cross_encoder_fallback"` 区分清晰

### 架构集成
- Reranker 管道流程清晰：RRF → boost → cross-encoder → diversity
- `cross_encoder_enabled: false` 默认关闭是正确的渐进发布策略
- 配置项 `cross_encoder_model`, `cross_encoder_device` 暴露合理
