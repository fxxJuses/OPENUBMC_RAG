## 4. 根因分析

### 为什么重建索引后反而更差？

上一轮的不对称评估中，旧索引只有子 token（`get`, `sensor`, `data`），查询虽然有额外的复合 token（`getsensordata`），但子 token 仍能匹配，所以只产生少量噪音。

**重建索引后，索引端也使用增强分词器，产生了大量复合 token + 子 token。** 这导致：

1. **BM25 词频膨胀**：每个代码标识符产生多个 token（原始 + 子 token），文档长度显著增加，BM25 的长度归一化（b 参数）对长文档施加惩罚
2. **IDF 权重稀释**：常见子 token（如 `get`, `set`, `data`, `value`）因出现在更多文档中而被进一步压低 IDF，而领域 token 的区分度没有等比例提升
3. **TF 噪音**：复合 token（如 `getsensordata`）在查询中很少直接匹配用户意图，但占据了 BM25 的匹配名额

具体证据：
- BM25 索引文件从 8.5MB 增长到 15.9MB（接近翻倍），说明 token 数量大幅增加
- `single_component` 类别退化最严重（-21.4pp），这类查询通常针对单个组件的函数/方法，代码分词的噪音对精确匹配影响最大
- `cross_component` 没有继续恶化（维持 0.5882），因为跨组件查询本来就依赖语义而非精确匹配

## 6. 建议

1. **回退 BM25 索引**：恢复旧索引（`data/index.pre-p3.bak`），确认回退后指标恢复基线
2. **调优分词策略**：
   - 考虑仅保留子 token（不保留复合 token），避免词频膨胀
   - 或调整 BM25 参数（降低 b 值减弱长度归一化）
   - 或对增强分词结果做去重/去常见 token 过滤
3. **领域词典可保留**：领域术语（IPMI/SEL/SDR）的注入应该是有益的，问题在于 camelCase/snake_case 拆分产生的复合 token
